Resource Locks —
beskytt mot utilsiktet sletting

Satt opp Resource Locks og DenyAction-policy for å beskytte lab-miljøet mot utilsiktet sletting. Kombinerer manuell lock på kritiske ressurser med en policy som blokkerer sletting av alle ressursgrupper.

Fullført Resource Locks DenyAction Governance AZ-104 pensum

Hva er Resource Locks og hvorfor bruker vi det?

Resource Locks hindrer at ressurser i Azure blir slettet eller endret ved en feil — selv av brukere med Owner-rolle. En lock overstyrer alle RBAC-tillatelser. Det er den siste forsvarslinjen mot menneskelige feil.

Delete
Ressursen kan leses og endres, men ikke slettes. Den vanligste lock-typen i produksjon — beskytter mot utilsiktet sletting uten å hindre drift.
ReadOnly
Ressursen kan kun leses — ingen endringer, ingen sletting. Brukes sjeldnere siden det blokkerer normal drift. Typisk brukt under frysperioder eller revisjoner.
Arv nedover
En lock på ressursgruppe-nivå arves av alle ressurser inni. Du trenger ikke sette lock på hver enkelt ressurs.
Hvem kan fjerne den?
Kun brukere med Microsoft.Authorization/locks/delete-rettighet — typisk Owner eller User Access Administrator. Contributor kan IKKE fjerne locks.

Hvorfor ikke ReadOnly? ReadOnly ville blokkert normal lab-bruk — du kan ikke starte/stoppe VM-er, endre NSG-regler eller oppdatere tags. Delete gir beskyttelse uten å hindre drift.

Steg 4 — DenyAction policy Fullført
1
Hvorfor DenyAction er bedre enn locks for ressursgrupper
Det ble forsøkt å sette opp en deployIfNotExists-policy som automatisk deployer locks på nye ressursgrupper — men dette er teknisk krevende fordi locks på ressursgrupper krever subscription-scope ARM-template deployment. DenyAction-effekten løser det samme problemet på en enklere måte.
Resource Lock
Owner kan fjerne. Krever manuell oppsett per ressurs. Arves nedover fra ressursgruppe.
DenyAction
Contributor kan ikke fjerne. Ingen Managed Identity. Blokkerer direkte i policy-laget. Enklere å skalere.
2
Opprett og tilordne DenyAction-policy
En custom policy-definisjon ble opprettet med denyAction-effekt som blokkerer delete-operasjoner på alle ressursgrupper i subskripsjonen.
"mode": "All", "policyRule": { "if": { "field": "type", "equals": "Microsoft.Resources/subscriptions/resourceGroups" }, "then": { "effect": "denyAction", "details": { "actionNames": ["delete"] } } }
InnstillingVerdiForklaring
NavnDeny delete on resource groups—
ScopeNordframe-Lab (subskripsjons-nivå)Gjelder alle ressursgrupper i subskripsjonen
EffektdenyActionBlokkerer delete-operasjoner direkte
Krever Managed IdentityNeiIngen deployment — bare blokkering
💡
Verifisert: Forsøk på å slette rg-lab-test resulterte i feilmelding: "Deletion of resource was disallowed by policy". DenyAction fungerer som forventet.
⚠️
For å slette en ressursgruppe mens DenyAction er aktiv må du midlertidig sette Policy enforcement: Disabled på assignmenten, slette ressursgruppen, og aktivere policyen igjen.
