Azure Storage Account —
blob, filer og tilgangsstyring

Hva er Resource Locks og hvorfor bruker vi det?

Resource Locks hindrer at ressurser i Azure blir slettet eller endret ved en feil — selv av brukere med Owner-rolle. En lock overstyrer alle RBAC-tillatelser. Det er den siste forsvarslinjen mot menneskelige feil.

Delete
Ressursen kan leses og endres, men ikke slettes. Den vanligste lock-typen i produksjon — beskytter mot utilsiktet sletting uten å hindre drift.
ReadOnly
Ressursen kan kun leses — ingen endringer, ingen sletting. Brukes sjeldnere siden det blokkerer normal drift. Typisk brukt under frysperioder eller revisjoner.
Arv nedover
En lock på ressursgruppe-nivå arves av alle ressurser inni. Du trenger ikke sette lock på hver enkelt ressurs.
Hvem kan fjerne den?
Kun brukere med Microsoft.Authorization/locks/delete-rettighet — typisk Owner eller User Access Administrator. Contributor kan IKKE fjerne locks.

InnstillingVerdiForklaring

Steg 4 — Tilgangsstyring og SAS-token Fullført
1
Generer et SAS-token for begrenset tilgang
Et SAS-token (Shared Access Signature) gir begrenset, tidsbegrenset tilgang til en spesifikk ressurs — uten å dele Access Key. Brukes når du vil gi en ekstern part midlertidig lesetilgang til én fil eller container.
InnstillingAnbefalt verdiForklaring
PermissionsReadKun lesing — aldri gi mer enn nødvendig
Start timeNå—
Expiry time1 time (lab)I produksjon: kortest mulig levetid
Allowed protocolsHTTPS onlyAldri HTTP
💡
Tre tilgangsmetoder — sammenligning:
Access Keys — full tilgang til alt. Del aldri. Roter jevnlig.
SAS-token — begrenset tilgang med utløpsdato. Bruk for ekstern deling.
RBAC — anbefalt for brukere og applikasjoner. Granulær og sporbar.

Opprett Storage Account i portalen
Gå til portal.azure.com → Create a resource → Storage account. Storage Account-navnene er globalt unike på tvers av hele Azure — kun lowercase bokstaver og tall, 3-24 tegn.
InnstillingVerdiForklaring
Resource grouprg-lab-01Arver tags og policies
Storage account namestnflab01Globalt unikt, kun lowercase + tall, ingen bindestrek
RegionNorway EastSamme som resten av lab-miljøet
PerformanceStandardPremium gir lavere latency — ikke nødvendig i lab

Steg 3 — SAS-token og tilgangsstyring Fullført
1
Generer et SAS-token for blob-filen
SAS (Shared Access Signature) er en tidsbegrenset URL med spesifikke tillatelser. I stedet for å gi noen full tilgang til Storage Account gir du dem kun tilgang til én fil i én time.
💡
Gå til: stnflab01 → Containers → lab-filer → klikk på filen → Generate SAS
InnstillingVerdiForklaring
PermissionsReadKun lese — ikke skrive eller slette
Expiry1 time fra nåKorteste praktiske varighet for testing
Allowed protocolsHTTPS onlyAlltid — aldri HTTP i produksjon
⚠️
SAS-tokens kan ikke tilbakekalles (med mindre du roterer Access Key). I produksjon bør du bruke User Delegation SAS (basert på Entra ID) fremfor Service SAS (basert på Access Key) — da kan du tilbakekalle ved å fjerne tilgang i Entra ID.
2
RBAC-tilgang til Storage Account
For brukere og applikasjoner er RBAC bedre enn SAS-tokens — tilgangen kan tilbakekalles når som helst, og alt logges i Entra ID.
RolleHva den girTypisk bruker
Storage Blob Data OwnerFull tilgang til blob-dataAdmin, scripts
Storage Blob Data ContributorLes, skriv og slett blobsApplikasjoner som skriver data
Storage Blob Data ReaderKun lese blobsApplikasjoner som kun leser
Storage Account ContributorAdministrer storage — ikke dataIT-admin

1
Opprett File Share og koble til som nettverksdisk
Azure Files lar deg montere en nettverksdisk i Windows akkurat som en vanlig filserver — via SMB-protokollen. Gå til stnflab01 → Data storage → File shares → + File share.
InnstillingVerdiForklaring
Namelab-shareKun lowercase og bindestrek
TierTransaction optimizedBilligste alternativ — greit for lab. Premium gir lavere latency
💡
Klikk på lab-share → Connect for å få ferdig PowerShell-script som monterer disken på Windows. Scriptet inneholder Access Key automatisk — kjør det på vm-lab-01 for å montere som Z:\ eller annen bokstav.
⚠️
Port 445 (SMB) må være åpen mot Azure. I noen bedriftsnettverk er denne porten blokkert av brannmur — da fungerer ikke montering fra on-premises. Fra Azure VM-er fungerer det alltid.
