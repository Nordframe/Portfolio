blob, filer og tilgangsstyring

Hva er Resource Locks og hvorfor bruker vi det?

Resource Locks hindrer at ressurser i Azure blir slettet eller endret ved en feil — selv av brukere med Owner-rolle. En lock overstyrer alle RBAC-tillatelser. Det er den siste forsvarslinjen mot menneskelige feil.

Delete
Ressursen kan leses og endres, men ikke slettes. Den vanligste lock-typen i produksjon — beskytter mot utilsiktet sletting uten å hindre drift.
ReadOnly
Ressursen kan kun leses — ingen endringer, ingen sletting. Brukes sjeldnere siden det blokkerer normal drift. Typisk brukt under frysperioder eller revisjoner.
Arv nedover
En lock på ressursgruppe-nivå arves av alle ressurser inni. Du trenger ikke sette lock på hver enkelt ressurs.
Hvem kan fjerne den?
Kun brukere med Microsoft.Authorization/locks/delete-rettighet — typisk Owner eller User Access Administrator. Contributor kan IKKE fjerne locks.

lab-share brukes også av Cloud Shell — Azure Cloud Shell lagrer sine filer i samme File Share under mappen .cloudconsole. Verifiser filinnhold via File shares → lab-share → Browse i portalen.
2
PowerShell — monter File Share
Kjørt fra Cloud Shell (PowerShell). Hent Access Key som variabel for å unngå eksponering.
# Lagre Access Key som variabel — vises ikke i terminal
$key = az storage account keys list --resource-group rg-lab-01 --account-name stnflab01 --query '[0].value' -o tsv

# Test port 445 (SMB)
Test-NetConnection -ComputerName stnflab01.file.core.windows.net -Port 445

# Monter som Z:\
New-PSDrive -Name Z -PSProvider FileSystem -Root "\stnflab01.file.core.windows.net\lab-share" -Credential (New-Object System.Management.Automation.PSCredential("localhost\stnflab01", (ConvertTo-SecureString $key -AsPlainText -Force))) -Persist

# Verifiser
Get-ChildItem Z:
⚠️
Port 445 (SMB) er blokkert i mange bedriftsnettverk. Fra Azure VM-er og Cloud Shell fungerer det alltid. Fra on-premises kreves VPN eller ExpressRoute.
